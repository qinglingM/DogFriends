# PRD：去玩 Tab · 宠物友好地点模块

## 1. 模块定位

### 1.1 模块名称

**去玩**

### 1.2 所属位置

APP 底部四个一级 Tab 中的一个：

| Tab | 作用            |
| --- | ------------- |
| 遛狗  | 记录遛狗轨迹、时长、距离  |
| 去玩  | 发现城市里适合带狗去的地方 |
| 记录  | 查看历史遛狗/出行记录   |
| 我的  | 用户资料、狗狗资料、设置  |

当前 PRD 只描述 **「去玩」Tab**。

### 1.3 一句话定义

**「去玩」是一个帮助狗主人发现当前城市内宠物友好地点的列表型功能模块。**

用户在周末或日常想带狗出门时，可以打开「去玩」，查看这个城市里有哪些公园、咖啡店、餐厅适合带狗去，并查看每个地点的宠物友好规则、最近验证状态，也可以新增自己发现的宠物友好地点。

### 1.4 核心用户场景

用户不是为了“查附近 500 米有什么”，而是为了：

> “我想带狗在这个城市里找个地方玩，哪里可以去？”

典型场景：

| 场景    | 用户问题               |
| ----- | ------------------ |
| 周末出门  | 这个城市有哪些地方适合带狗去？    |
| 想喝咖啡  | 哪些咖啡店可以带狗？         |
| 想逛公园  | 哪些公园对狗友好？          |
| 想带狗吃饭 | 哪些餐厅能带狗？           |
| 发现新地点 | 我知道一个宠物友好地点，想提交给大家 |
| 信息更新  | 我去了之后发现规则变了，想更新信息  |

# 3. 信息源与数据原则

## 3.1 地点底库来源

地点底库可以来自地图 POI 服务，例如高德、腾讯位置服务、百度地图等。高德地点搜索支持关键字、周边、多边形区域、ID 等检索方式；腾讯位置服务支持区域、周边、矩形、多边形范围和地点详情；百度地图也提供地点检索、周边检索、地点详情等能力。POI 数据适合提供地点名称、地址、电话、坐标、分类等基础信息，但不能直接判断“是否宠物友好”。([Wikipedia][1])

## 3.2 宠物友好信息来源

宠物友好信息来自：

| 来源      | 用途         |
| ------- | ---------- |
| 平台运营预置  | 初始种子地点     |
| C 端用户新增 | 补充新地点      |
| C 端用户验证 | 更新地点规则     |
| 用户上传图片  | 证明现场情况     |
| 用户反馈冲突  | 触发平台审核     |
| 后续商家认领  | 后续版本，不属于本期 |

## 3.3 数据可信原则

地点是否宠物友好，不能只存一个“是/否”，必须存规则。

核心规则包括：

| 字段     | 示例                        |
| ------ | ------------------------- |
| 是否可带狗  | 可带 / 仅户外 / 不稳定 / 不建议      |
| 可进入区域  | 室内 / 室外 / 仅露台 / 仅门口       |
| 适合狗狗类型 | 小型犬 / 中型犬 / 大型犬 / 所有体型    |
| 行为要求   | 牵绳 / 嘴套 / 不上椅 / 不吠叫       |
| 配套设施   | 水碗 / 宠物餐 / 拾便袋 / 遮阳区      |
| 最近验证时间 | 2 天前 / 7 天前 / 30 天前       |
| 验证状态   | 已验证 / 待审核 / 信息更新中 / 审核未通过 |

# 4. 页面结构总览

本模块共包含 8 个核心页面/状态：

```text
去玩首页
  ├─ 新增宠物友好地点入口
  ├─ 地点列表
  ├─ 搜索与筛选
  └─ 点击地点进入详情

地点详情页
  ├─ 查看宠物规则
  ├─ 收藏 / 导航 / 电话
  └─ 我去过，更新信息

更新信息页
  └─ 提交已有地点的最新宠物友好状态

新增宠物友好地点流程
  ├─ 选择新增方式页
  ├─ 填写地点基本信息页
  ├─ 填写宠物友好规则页
  ├─ 上传图片与补充信息页
  ├─ 提交成功页
  └─ 提交记录 / 审核通知
```

## 8.4 狗狗可进入区域

字段名：

```text
宠物可进入区域 *
```

选项：

| 选项     | 含义                   |
| ------ | -------------------- |
| 室内可进   | 狗狗可以进入室内             |
| 室外可进   | 狗狗可以进入户外区域           |
| 仅露台/户外 | 只能坐露台或户外             |
| 指定区域可进 | 只能进入指定区域             |
| 不可进入   | 用户想提交“其实不可带”的纠错信息时使用 |

本字段单选或多选均可。建议本期使用单选，避免信息冲突。

推荐默认逻辑：

| 用户选择   | 系统状态           |
| ------ | -------------- |
| 室内可进   | 宠物状态偏强         |
| 仅露台/户外 | 标记为“仅户外”       |
| 不可进入   | 标记为“待审核/可能不推荐” |

# 17. 审核规则

## 17.1 新增地点审核

新增地点提交后默认进入：

```text
待审核
```

平台审核通过后，才进入去玩列表正式展示。

## 17.2 自动进入人工审核的情况

| 情况        | 原因        |
| --------- | --------- |
| 新增地点      | 防止虚假地点    |
| 上传图片      | 需要判断图片有效性 |
| 标记“不可进入”  | 可能影响地点展示  |
| 同地点信息冲突   | 需要人工判断    |
| 用户举报信息有误  | 需要复核      |
| 用户提交大型犬限制 | 对用户决策影响大  |

## 17.3 审核结果

| 结果      | 用户看到                 |
| ------- | -------------------- |
| 审核通过    | 地点展示在去玩列表            |
| 审核未通过   | 消息通知原因，可修改           |
| 需要补充    | 进入提交记录页补充            |
| 合并到已有地点 | 提醒用户该地点已存在，信息已作为更新提交 |

# 23. 最终 PRD 结论

本模块不是一个独立平台，而是整个遛狗 APP 中的一个轻量一级 Tab。

它只承担一件事：

> **当用户想带狗出去玩时，告诉他这个城市里哪些地方可以去，并让用户共同维护这些地点是否真的宠物友好。**

本期最重要的体验是：

1. **打开去玩，能看到当前城市适合带狗去的地点；**
2. **点进地点，能看懂狗能不能去、能进哪里、适合什么狗；**
3. **用户去过后，可以很轻地更新信息；**
4. **用户发现新地点，可以提交新增宠物友好地点；**
5. **新增地点审核通过后，进入城市宠物友好地点列表。**
