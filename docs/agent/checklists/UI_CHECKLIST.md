# UI Checklist

本文件用于处理界面、交互、输入、头像、安全区、间距和状态反馈相关任务。

## 一、输入键盘规则

所有输入框都要根据场景判断键盘右下角按钮的功能，不能默认全部是换行。

常见规则：

- 搜索框：键盘按钮应为“搜索”
- 评论框：键盘按钮可为“发送”或“换行”，根据产品需求判断
- 昵称 / 签名 / 资料编辑：键盘按钮应为“完成”
- 多行文本：可以保留“换行”
- 表单最后一项：可以考虑“完成”

检查点：

- 输入栏是否被键盘遮挡
- 评论输入框是否悬浮在键盘上方
- 页面滚动时输入内容是否可见
- 键盘弹出后底部按钮是否仍可点击
- 不同系统的键盘表现是否一致
- 搜索、完成、发送、换行是否符合当前场景

## 二、头像规则

所有上传图片内容的功能，都必须检查头像形状与裁剪功能是否匹配。

检查点：

- 当前头像是圆形还是方形
- 裁剪框是否与展示形状一致
- 上传后是否立即刷新
- 刷新页面后是否仍然统一
- 不同页面是否使用同一个头像源
- 头像缓存是否会导致旧图残留
- 默认头像是否统一
- 上传失败是否有提示

头像使用场景包括但不限于：

- 个人主页
- 资料编辑页
- 内容流
- 评论区
- 点赞列表
- 收藏列表
- 消息通知
- 搜索结果
- 宠物主页 / 狗狗档案

## 三、安全区规则

苹果和安卓的安全区上下边界必须谨慎处理。

重点检查：

- 顶部是否被刘海、灵动岛、状态栏遮挡
- 底部按钮是否被 Home Indicator 遮挡
- 安卓底部导航栏是否遮挡内容
- 弹窗、输入框、底部操作栏是否留出余量
- 滚动页面底部是否有足够 padding
- 是否存在不同机型错位问题

原则：

- 不要随意用绝对定位硬顶安全区。
- 优先使用安全区、布局容器和统一间距规则解决。
- 涉及底部操作按钮时，要额外检查 iOS 和安卓真机效果。

## 四、常见字段展示

常见字段必须统一判断是否需要存在、在哪里展示、如何刷新。

重点字段：

- 用户名称
- 用户头像
- 个性签名
- 城市
- 性别
- 点赞数
- 收藏数
- 评论数
- 发布状态
- 删除状态
- 审核状态
- 创建时间
- 更新时间

每个字段都要检查：

- 是否真的有必要展示
- 展示在哪些页面
- 来源是哪张表
- 修改后是否实时同步
- 刷新后是否一致
- 是否有默认值
- 是否允许为空
- 是否需要截断

## 五、间距与布局规范

AI 修改间距时，优先使用：

- gap
- padding
- margin
- 主题常量
- 统一 spacing token

不建议随意写大量零散数字。

如果某个间距是为了视觉效果人工微调，应在代码中添加注释。

示例：

```text
这里额外增加 4px，是为了让头像与昵称在视觉上居中，而不是严格数学居中。
```

绝对定位适合：

- 角标
- 浮层
- 悬浮按钮
- 图片上的覆盖信息
- 特殊动效层

不适合用绝对定位解决普通排版问题。

必要时用字符画确认结构：

```text
[头像]  昵称
        签名
        城市 · 性别
```

## 六、状态反馈规范

以下操作成功或失败时，最好都有文本提示：

- 发布
- 删除
- 保存
- 上传图片
- 修改资料
- 登录
- 注册
- 退出登录
- 点赞
- 收藏
- 评论
- 搜索失败
- 加载失败

错误不能导致：

- 白屏
- 崩溃
- 无反应
- 用户不知道是否成功
- 用户已输入内容丢失

错误场景至少应具备：

- 文本提示
- 重试按钮
- 返回入口
- 保留用户已输入内容
- 控制台可定位错误原因

## 七、防重复提交

以下状态必须防重复点击：

- 发布中
- 删除中
- 保存中
- 上传中
- 登录中
- 注册中
- 评论发送中

否则容易造成重复数据、重复评论、重复请求、数据库异常。
