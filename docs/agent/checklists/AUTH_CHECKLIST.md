# Auth Checklist

本文件用于处理账号注册、登录、找回、退出、注销和账号体系复杂度判断。

## 一、先判断账号体系复杂度

账号体系不能盲目堆功能。开工前需要先确认：

- 是否必须登录才能使用
- 是否允许游客浏览
- 是否允许游客发布
- 是否需要手机号
- 是否需要密码
- 是否需要验证码
- 是否需要第三方登录
- 是否涉及实名
- 是否涉及注销
- 是否涉及用户协议和隐私政策
- 是否涉及多端登录
- 是否涉及账号安全风险

## 二、基础账号功能

常规账号体系至少应考虑：

- 注册
- 登录
- 退出登录
- 忘记密码
- 修改密码
- 注销账号
- 用户协议
- 隐私政策

## 三、注册流程检查

注册流程需要检查：

- 注册方式是什么
- 手机号是否必填
- 密码是否必填
- 验证码是否必填
- 昵称是否注册时填写
- 是否需要同意用户协议和隐私政策
- 注册失败是否有明确提示
- 手机号已注册时如何提示
- 密码规则是否明确
- 注册成功后跳转哪里

## 四、登录流程检查

登录流程需要检查：

- 登录方式是什么
- 密码登录和验证码登录是否都支持
- 登录失败是否有明确提示
- 密码错误如何提示
- 账号不存在如何提示
- 网络错误如何提示
- 登录中是否禁止重复点击
- 登录成功后跳转哪里
- 登录状态是否持久化
- App 重启后是否仍然登录

## 五、忘记 / 修改密码

需要判断：

- 忘记密码是否必须做
- 修改密码入口在哪里
- 是否需要旧密码
- 是否需要验证码
- 修改成功后是否退出登录
- 修改失败是否有明确提示
- 密码规则是否统一

## 六、退出登录

退出登录需要检查：

- 入口在哪里
- 是否需要二次确认
- 退出后跳转哪里
- 本地缓存是否清理
- 用户数据是否仍然安全
- 再次打开 App 是否保持退出状态
- 是否影响游客浏览

## 七、注销账号

账号注销功能必须存在，尤其是面向正式上线的 App。

注销功能需要明确：

- 入口在哪里
- 是否二次确认
- 注销后数据如何处理
- 是否有冷静期
- 是否可以恢复
- 头像、内容、评论、点赞如何处理
- 注销后原手机号是否允许重新注册
- 注销是否影响数据库关联数据
- 是否符合应用市场审核要求

## 八、登录方式版本计划

登录方式可以分阶段：

- MVP：手机号 + 验证码 / 密码
- 后续：微信登录、Apple 登录
- 再后续：更多第三方登录

不能一开始为了完整而过度复杂，也不能完全没有拓展性。

## 九、账号相关风险

涉及以下内容时，必须先说明风险：

- 用户身份判断
- 权限判断
- 注销逻辑
- 删除用户数据
- 正式环境账号数据
- 第三方登录绑定
- 用户协议 / 隐私政策
- 手机号、验证码、密码等敏感信息
