# Release Checklist

本文件用于处理 Git、构建、部署、服务器、环境变量、数据库迁移和上线相关任务。

## 一、Git 操作规则

开工前必须明确当前 Git 工作方式：

- 当前分支是什么
- 是否允许直接改主分支
- 是否需要新建功能分支
- 提交前是否需要运行构建或测试
- 是否需要用户确认后再提交
- 是否允许 AI 自动 commit
- 是否允许 AI 自动 push
- commit message 使用什么格式

默认规则：

- 不明确时，不自动 push。
- 不明确时，不直接改主分支。
- 大改动前先说明修改范围。
- 涉及数据库、认证、支付、删除、权限、安全的改动，必须先说明风险。
- 每次提交应说明：改了什么、为什么改、影响范围是什么。

推荐 commit message 格式：

```text
fix: 修复 xxx 问题
feat: 新增 xxx 功能
refactor: 重构 xxx 逻辑
style: 调整 xxx 样式
docs: 更新 xxx 文档
chore: 调整构建/配置/依赖
```

## 二、构建前检查

构建前需要检查：

- 当前环境是本地、测试还是正式
- 依赖是否安装
- 环境变量是否完整
- 是否存在本地专用配置
- 是否有未提交修改
- 是否有 TypeScript / lint 报错
- 是否有构建缓存问题
- 是否需要清理缓存后重建

## 三、服务器环境检查

开工前必须明确：

- 当前是否有服务器
- 使用的是本地开发环境、测试环境还是正式环境
- 接口地址来自哪里
- 环境变量在哪里配置
- 哪些密钥不能暴露到前端
- 是否存在 CDN、对象存储、反向代理、云函数等服务

涉及服务器相关问题时，不能在缺少环境信息的情况下直接判断。

## 四、数据库环境检查

开工前必须明确：

- 使用的数据库类型
- 数据表结构
- 字段含义
- 用户身份字段
- 数据归属字段
- 是否有 RLS / 权限策略
- 是否有软删除或硬删除
- 是否有缓存或前端状态同步
- 是否需要考虑历史数据兼容

涉及数据库修改时，必须特别说明：

- 修改了哪张表
- 修改了哪些字段
- 是否影响旧数据
- 是否需要迁移
- 是否需要回滚方案

## 五、环境变量检查

需要明确：

- 哪些变量用于本地环境
- 哪些变量用于测试环境
- 哪些变量用于正式环境
- 哪些变量可以暴露到前端
- 哪些变量绝对不能暴露
- 是否有 `.env.example`
- 构建平台是否配置了相同变量
- 变量名是否统一

特别注意：

- API Key 不等于都可以暴露。
- 前端可见变量必须确认服务商是否允许公开。
- 正式环境变量不能写死在代码里。

## 六、上线 / 发布前检查

上线前需要检查：

- 构建是否成功
- 核心流程是否可用
- 登录是否正常
- 发布是否正常
- 删除是否正常
- 图片上传是否正常
- 数据刷新是否正常
- 安全区是否正常
- 正式环境接口是否正确
- 数据库权限是否正确
- 是否有明显控制台报错
- 是否有回滚方案

## 七、白屏 / 崩溃排查

遇到白屏或崩溃，优先检查：

- 控制台报错
- 构建日志
- 环境变量缺失
- 路由配置
- 资源路径
- API 请求失败
- 权限策略
- 代码签名
- 平台限制
- 缓存问题

不能只说“可能是网络问题”，必须给出可验证路径。

## 八、回滚规则

涉及正式环境时，必须考虑回滚：

- 是否可以回退到上一版本
- 是否有数据库迁移
- 数据迁移是否可逆
- 是否有备份
- 是否影响线上用户
- 是否需要暂停发布
