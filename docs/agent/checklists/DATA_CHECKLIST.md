# Data Checklist

本文件用于处理数据归属、刷新、删除、硬编码、状态同步和数据库一致性问题。

## 一、数据归属权规则

代码中的数据归属必须特别谨慎。

容易混淆的字段：

- userId
- username
- profileId
- profileName
- authorId
- ownerId
- petId
- dogId
- postId
- commentId

判断数据归属时，先回答：

- 这条数据是谁创建的？
- 这条数据属于谁？
- 谁可以编辑？
- 谁可以删除？
- 展示时应该显示谁的信息？
- 数据库查询应该按哪个 ID 过滤？

判断原则：

```text
ID 是房产证，name 是门牌号。判断归属时看 ID，不要看 name。
```

## 二、发布路径规则

发布功能必须保证路径清晰。

检查点：

- 从哪里进入发布
- 发布前是否需要登录
- 必填项是什么
- 图片是否可选
- 发布中是否显示状态
- 发布中是否禁止重复点击
- 发布成功后跳转哪里
- 发布失败后是否保留已填写内容
- 发布后列表是否刷新
- 发布后数据库是否正确写入

发布流程不能让用户产生“我到底发没发出去”的不确定感。

## 三、删除功能规则

删除功能必须稳定，不能只做表面删除。

需要明确：

- 是软删除还是硬删除
- 删除后列表是否立即消失
- 刷新后是否仍然消失
- 关联数据是否一起处理
- 评论、点赞、收藏是否需要同步删除或隐藏
- 删除失败是否有提示
- 删除中是否禁止重复点击
- 是否需要二次确认

涉及用户内容删除时，必须保证删除逻辑彻底、清晰、可验证。

## 四、点赞、收藏、评论字段

点赞、收藏、评论等字段必须统一处理。

检查点：

- 数量是否准确
- 用户操作后是否立即反馈
- 刷新后是否保持一致
- 是否存在重复点赞 / 重复收藏
- 取消操作后数量是否正确减少
- 前端状态和数据库状态是否一致
- 列表页和详情页是否同步
- 失败后是否回滚 UI 状态

## 五、刷新与重置规则

每个列表或页面都要明确刷新方式：

- 下拉刷新
- 点击刷新按钮
- 发布后自动刷新
- 删除后局部刷新
- 切换 Tab 后刷新
- 返回页面后刷新
- App 重新进入后刷新

涉及复杂刷新逻辑时，应在代码中注释说明：

- 为什么这里要刷新
- 刷新的数据范围是什么
- 是否会重置滚动位置
- 是否会清空用户输入
- 是否会影响分页
- 是否会触发重复请求

刷新、推送、推荐算法不要过早追求复杂。优先目标是：

- 逻辑清晰
- 可理解
- 可维护
- 可验证
- 不产生重复数据
- 不导致明显错乱

## 六、必填、最少、最多规则

几乎所有用户填写内容都要明确限制。

需要定义：

- 是否必填
- 最少多少字
- 最多多少字
- 是否允许空格
- 是否允许纯表情
- 是否允许特殊符号
- 是否允许重复名称
- 超出长度如何处理
- 前端限制还是后端限制
- 数据库存储长度是多少

常见参考：

- 昵称：2-12 个字符
- 签名：0-50 个字符
- 评论：1-300 个字符
- 标题：2-30 个字符
- 正文：1-2000 个字符

实际长度应根据具体产品调整。

## 七、时间显示规则

不同功能下的时间展示需要统一或有明确差异。

需要确认：

- 是否显示年月日
- 是否显示小时分钟
- 是否显示“刚刚”
- 是否显示“今天”
- 是否显示“昨天”
- 超过多久显示具体日期
- 是否需要显示年份
- 是否需要考虑时区
- 是否需要与服务端时间统一

推荐规则：

- 1 分钟内：刚刚
- 1 小时内：xx 分钟前
- 24 小时内：xx 小时前
- 昨天：昨天 HH:mm
- 今年内：MM-DD HH:mm
- 非今年：YYYY-MM-DD

具体项目可以根据产品风格调整，但规则必须统一。

## 八、硬编码检查规范

提到硬编码数据时，必须特殊标注。

需要检查：

- 数据写死在哪里
- 是否应该来自数据库
- 是否应该来自配置文件
- 是否应该来自环境变量
- 是否应该来自接口
- 是否会影响不同用户
- 是否会影响正式环境
- 是否会影响后续扩展

尤其注意：

- 用户 ID 不能硬编码
- 城市不能随意硬编码
- 服务器地址不能写死在正式代码里
- 密钥不能暴露在前端
- 测试数据不能混入正式环境
- 枚举值需要统一来源

硬编码不是绝对不能用，但必须知道它为什么存在、存在多久、未来怎么替换。
