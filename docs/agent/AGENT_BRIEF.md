# Agent 开工短规则

本文件是每次与 AI / Agent 协作前的必读短规则。  
它只保留最高频、最高价值的沟通规则，避免每次全量读取长文档导致 token 浪费。

## 一、默认沟通规则

1. 默认使用中文回答。
2. 先判断用户问题类型，再决定输出结构，不要对所有回答强制套用统一模板。
3. 只有在任务型、改动型、判断型问题中，才需要判断是否存在 P0-P5 优先级。
4. 简单信息问答、闲聊式反馈、进度确认类问题，不强制输出优先级。
5. 只有在开发、修改、排查、验收相关任务中，才默认提供可操作的验收步骤。
6. 默认不要输出完整代码，只说明修改文件、修改点、原因、风险和验证方式。
7. 除非用户明确要求，或关键代码对理解问题非常必要，否则只展示最相关代码片段。
8. 关键问题缺少信息时必须先问，不要擅自判断。
9. 如果用户要求违反了之前设定的规则，必须及时提醒。
10. 解释复杂问题时可以多用类比，但不要为了类比而废话。
11. 修改范围、风险点、验收步骤只在相关任务中输出；不适用时省略，不要硬套。
12. 回答应只覆盖与用户问题直接相关的内容，不要为了完整而补充无关栏目。
13. 完成开发、修改、排查类任务后，必须说明：优先级、修改范围、风险点、验收步骤。
14. 展示结果时，优先使用表格整理关键信息，尤其适用于“改前 / 改后 / 原因 / 影响”这类对照内容。
15. 除非用户明确要求，或文件列表本身对理解问题非常关键，否则不要把所有修改文件逐个罗列给用户。
16. 说明改动结果时，优先讲“改前是什么样、改后是什么样、为什么这样改”，并尽量用表格表达。

## 二、关键风险规则

涉及以下内容时，必须先说明风险，不要直接动手：

- 数据库结构
- 用户权限
- 删除逻辑
- 账号体系
- 正式环境
- Git push / merge
- 服务器配置
- 支付或安全相关逻辑
- 可能影响线上用户的数据改动

## 三、按需读取规则

不要每次全量读取所有规则文件。  
根据本次任务类型，按需读取 `AGENT_CHECKLIST_INDEX.md` 中指定的专项 checklist。

除非任务明确需要，否则禁止全量读取所有 checklist。

## 四、默认输出格式

开发、修改、排查类问题默认按以下格式输出：

```text
结论：
问题优先级：P?
原因：xxx

修改范围：
1. xxx
2. xxx

风险点：
1. xxx
2. xxx

验收步骤：
1. xxx
2. xxx
3. xxx

需要用户确认：
1. xxx
```

如果是极小的开发类问题，可以简化，但不能省略优先级和验收方式。

如果适合对照展示，优先使用表格，例如：

```text
| 项目 | 改前 | 改后 | 原因 |
|---|---|---|---|
| 评论输入 | 键盘遮挡 | 跟随键盘上移 | 保证输入可见 |
| 超长提示 | 显示字数计数 | 超限时 2 秒提示 | 减少技术打扰 |
```

默认不要额外列出全部修改文件；只在以下情况列出：

- 用户明确要求
- 文件范围本身就是讨论重点
- 不列出来会影响理解或验收

简单问答、反馈确认、进度同步类问题：

```text
直接回答与问题相关的内容即可。
如果不涉及改动判断，不强制输出：
- P0-P5
- 修改范围
- 风险点
- 验收步骤
```

示例：

- “现在这个功能比以前好多了，你觉得呢？” -> 直接回答看法，不需要优先级。
- “我们都做到哪儿了？” -> 直接同步进度，不需要验收步骤。
- “这么改行不行？” -> 属于判断型问题，需要判断是否存在优先级。
