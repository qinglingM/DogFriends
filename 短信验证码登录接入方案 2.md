# 短信验证码登录接入方案

> 适用范围：将“食鉴”当前的短信验证码登录能力迁移到另一个 Web、H5 或 Capacitor 项目。  
> 当前实现核对日期：2026-06-28。  
> 技术栈：Supabase Auth + Supabase Edge Functions + 阿里云号码认证服务。

## 1. 接入完成后的能力

按本文接入后，新项目将具备：

- 中国大陆手机号短信验证码登录；
- 未注册手机号首次验证码登录时自动创建账号；
- 手机号 + 密码登录；
- 短信验证码验证后重置密码；
- Supabase Session 持久化、自动刷新及登录态恢复；
- 新用户创建后自动同步业务侧 `profiles` 用户资料；
- 客户端 60 秒重发倒计时，以及服务端按手机号/IP 的基础限流。

当前产品没有独立的注册页面。验证码登录同时承担注册功能：

```text
手机号 → 发送验证码 → 校验验证码
                         ├─ 已有用户：签发 Session
                         └─ 新用户：创建 Auth 用户并签发 Session
```

## 2. 当前项目采用的真实方案

当前项目没有使用 Supabase 内置短信供应商直接发 OTP。实际链路如下：

1. 前端调用公开的 Supabase Edge Function `sms-otp`；
2. Edge Function 生成 6 位验证码；
3. Edge Function 使用阿里云 `dypnsapi` 的 `SendSmsVerifyCode` 接口发送短信；
4. Edge Function 返回一个 5 分钟有效的、HMAC 签名的 OTP token；
5. 前端提交手机号、验证码和 OTP token；
6. Edge Function 校验验证码，通过 Supabase `service_role` 创建或更新用户；
7. Edge Function 返回 Supabase `access_token` 和 `refresh_token`；
8. 前端调用 `supabase.auth.setSession()` 建立登录态。

因此，阿里云 AccessKey 和 Supabase `service_role` 只允许放在 Edge Function Secret 中，绝不能放入前端环境变量或打包产物。

## 4. 必需的账号与服务

### 4.1 Supabase

准备一个 Supabase 项目，并取得：

- Project URL；
- Anon Key 或 Publishable Key；
- Service Role Key 或新密钥体系中的 Secret Key；
- Project Ref。

当前仓库关联的 Project Ref 只属于食鉴项目，新项目必须使用自己的 Project Ref。

### 4.2 阿里云

开通阿里云号码认证服务，并完成：

- 创建一个只允许调用所需短信接口的 RAM 用户；
- 创建 AccessKey ID 和 AccessKey Secret；
- 创建并审核短信签名；
- 创建并审核验证码模板；
- 模板参数与当前函数保持一致：

```json
{
  "code": "6位验证码",
  "min": "5"
}
```

当前函数调用参数为：

| 参数 | 当前值 |
| --- | --- |
| API Endpoint | `https://dypnsapi.aliyuncs.com/` |
| Action | `SendSmsVerifyCode` |
| Version | `2017-05-25` |
| 请求方法 | `POST` |
| 签名算法 | `HMAC-SHA1` |
| 验证码长度 | 6 位 |
| OTP 有效期 | 5 分钟 |

如果新模板只声明了 `code`，应同步删除函数中的 `min` 参数；模板参数必须与阿里云审核通过的模板完全一致。

## 5. 环境变量和 Secret

### 5.1 前端环境变量

在新项目前端的 `.env.local` 和生产构建环境中配置：

```dotenv
VITE_SUPABASE_URL=https://<project-ref>.supabase.co
VITE_SUPABASE_ANON_KEY=<anon-key-or-publishable-key>
```

这两个值会进入浏览器 bundle，本身是公开客户端凭据。数据库安全必须依赖 RLS，不能依赖隐藏 Anon Key。

当前项目还存在 `VITE_SUPABASE_FUNCTIONS_URL`，但短信调用使用 `supabase.functions.invoke()`，并不直接读取这个变量，因此短信登录本身不要求配置它。

### 5.2 Edge Function Secrets

必须配置：

| Secret | 用途 |
| --- | --- |
| `ALIYUN_ACCESS_KEY_ID` | 阿里云 RAM AccessKey ID |
| `ALIYUN_ACCESS_KEY_SECRET` | 阿里云 RAM AccessKey Secret |
| `ALIYUN_SMS_SIGN_NAME_B64` | 短信签名的 UTF-8 Base64，推荐使用 |
| `ALIYUN_SMS_TEMPLATE_CODE` | 审核通过的模板编码 |
| `ALIYUN_OTP_HMAC_SECRET` | 对 OTP token 签名的独立随机密钥 |
| `SUPABASE_URL` | Supabase 项目 URL，云端通常自动注入 |
| `SUPABASE_ANON_KEY` | Supabase 客户端 Key，云端通常自动注入 |
| `SUPABASE_SERVICE_ROLE_KEY` | 管理 Auth 用户，云端通常自动注入 |

代码兼容明文 `ALIYUN_SMS_SIGN_NAME`，但会优先读取 `ALIYUN_SMS_SIGN_NAME_B64`。当前线上项目同时存在两个 Secret。新项目建议只以 Base64 值为权威，避免中文签名在 CLI 或 Secret 写入过程中出现编码问题。

生成配置值：

```bash
# macOS
printf %s '你的短信签名' | base64

# 生成独立的 OTP HMAC 密钥
openssl rand -base64 48
```

写入 Supabase Secrets：

```bash
supabase login
supabase link --project-ref <project-ref>

supabase secrets set \
  ALIYUN_ACCESS_KEY_ID='<access-key-id>' \
  ALIYUN_ACCESS_KEY_SECRET='<access-key-secret>' \
  ALIYUN_SMS_SIGN_NAME_B64='<base64-sign-name>' \
  ALIYUN_SMS_TEMPLATE_CODE='<template-code>' \
  ALIYUN_OTP_HMAC_SECRET='<random-secret>'
```

检查 Secret 名称是否齐全：

```bash
supabase secrets list
```

该命令只显示名称和摘要，不会返回 Secret 原文。不要把真实 Secret 写入仓库、截图、前端 `.env` 或本文。

## 6. 必须修改的项目标识

复制 `sms-otp/index.ts` 后，修改合成邮箱后缀：

```ts
return `phone-${digest.slice(0, 32)}@phone.your-domain.example`
```

其作用是为每个手机号确定性生成一个 Supabase Auth 邮箱账号：

```text
SHA-256("phone:" + E.164手机号)
→ 取前32位
→ phone-<hash>@phone.your-domain.example
```

要求：

- 域名在该项目生命周期内保持不变；
- 同一套已有用户数据中不能中途更换，否则同一手机号可能映射到不同邮箱；
- 该邮箱不是用户真实邮箱，不应向其发送邮件；
- 如果新项目需要同时保留真实邮箱，不应直接照搬当前“登录时把 email 更新为合成邮箱”的逻辑，应增加独立身份映射表。

同时将 CORS 从当前的通配符改为新项目允许的来源。当前代码是：

```ts
'Access-Control-Allow-Origin': '*'
```

Web 生产环境建议改成明确域名。若同时支持 Web、iOS 和 Android，应根据请求来源实现允许列表，并正确处理 Capacitor 的 origin。

## 8. 数据库与用户资料同步

### 8.1 只需要 Auth，不需要业务 profiles

如果新项目只使用 `auth.users`，可以不复制食鉴的 `profiles` 表和白名单迁移。`sms-otp` 登录本身只依赖 Supabase Auth。

### 8.2 需要业务 profiles

业务用户表最小结构可采用：

```sql
create table public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  nickname text not null,
  phone text unique,
  phone_verified_at timestamptz,
  created_at timestamptz not null default now()
);
```

创建新用户同步函数：

```sql
create or replace function public.handle_new_user()
returns trigger
language plpgsql
security definer
set search_path = public
as $$
begin
  insert into public.profiles (
    id,
    nickname,
    phone,
    phone_verified_at
  )
  values (
    new.id,
    coalesce(
      nullif(trim(new.raw_user_meta_data ->> 'nickname'), ''),
      case
        when length(regexp_replace(coalesce(new.phone, ''), '[^0-9]', '', 'g')) >= 4
          then '用户' || right(regexp_replace(new.phone, '[^0-9]', '', 'g'), 4)
        else '新用户'
      end
    ),
    nullif(trim(coalesce(new.phone, '')), ''),
    new.phone_confirmed_at
  )
  on conflict (id) do nothing;

  return new;
end;
$$;

drop trigger if exists on_auth_user_created on auth.users;
create trigger on_auth_user_created
  after insert on auth.users
  for each row execute function public.handle_new_user();
```

如果新项目完整继承食鉴数据库，则按原迁移顺序执行，最终以 `0012_phone_binding_exempt_mobiles.sql` 中的 `handle_new_user()` 为准。不要只执行 `0011` 后漏掉后续覆盖，也不要把食鉴的白名单手机号带入新项目。

## 10. 前端接口封装

复制 `web/src/features/auth/aliyunSmsOtp.ts`，或实现等价封装：

```ts
type SmsOtpResponse = {
  ok: boolean
  token?: string
  login_phone?: string
  access_token?: string
  refresh_token?: string
  expires_in?: number
  token_type?: string
  error?: string
}

async function callSmsOtp(body: Record<string, unknown>): Promise<SmsOtpResponse> {
  const { data, error } = await supabase.functions.invoke<SmsOtpResponse>(
    'sms-otp',
    { body },
  )

  if (error) return { ok: false, error: error.message }
  return data ?? { ok: false, error: '短信服务没有返回结果' }
}
```

手机号统一转换为 E.164：

```ts
export function normalizeChinaMobileToE164(input: string): string | null {
  const trimmed = input.trim().replace(/\s+/g, '')
  if (/^\+[1-9]\d{6,14}$/.test(trimmed)) return trimmed

  const digits = trimmed.replace(/\D/g, '')
  if (digits.length === 11 && digits.startsWith('1')) return `+86${digits}`
  if (digits.length === 13 && digits.startsWith('86')) {
    const rest = digits.slice(2)
    if (rest.length === 11 && rest.startsWith('1')) return `+86${rest}`
  }
  return null
}
```

当前短信网关仅按中国大陆号码处理：发往阿里云时会从 `+86xxxxxxxxxxx` 去掉 `+86`。虽然前端函数接受其他 E.164 格式，但 Edge Function 没有完整的国际短信适配。新项目需要国际号码时必须单独改造供应商参数和号码校验。

## 11. 三条业务流程

### 11.1 验证码登录，首次登录自动注册

发送：

```ts
const sent = await callSmsOtp({
  action: 'send',
  phone: '+8613800138000',
  purpose: 'login',
})

// sent.token 必须暂存在当前登录流程中
```

校验并登录：

```ts
const result = await callSmsOtp({
  action: 'complete-login',
  phone: '+8613800138000',
  code: '123456',
  token: sent.token,
})

if (!result.ok || !result.access_token || !result.refresh_token) {
  throw new Error(result.error ?? '验证码登录失败')
}

const { error } = await supabase.auth.setSession({
  access_token: result.access_token,
  refresh_token: result.refresh_token,
})
if (error) throw error
```

行为：

- 手机号不存在：创建用户，生成随机内部密码并立即返回 Session；
- 手机号已存在：规范化账号信息，通过服务端 magic link 机制换取 Session；
- Auth 用户的 `phone` 存为 E.164；
- `phone_confirm` 和 `email_confirm` 均被服务端置为已确认。

### 11.2 手机号 + 密码登录

```ts
const result = await callSmsOtp({
  action: 'password-login',
  phone: '+8613800138000',
  password: '<password>',
})

await supabase.auth.setSession({
  access_token: result.access_token!,
  refresh_token: result.refresh_token!,
})
```

当前验证码自动注册用户使用随机内部密码，因此用户只有在完成“忘记密码/设置密码”后，才知道可用于手机号密码登录的密码。

### 11.3 忘记密码

第一步，发送验证码。只有已经注册的手机号才能发送：

```ts
const sent = await callSmsOtp({
  action: 'send',
  phone,
  purpose: 'forgot',
})
```

第二步，可选地先验证，以便 UI 进入设置密码页面：

```ts
await callSmsOtp({
  action: 'verify',
  phone,
  code,
  token: sent.token,
})
```

第三步，设置密码并直接登录：

```ts
const result = await callSmsOtp({
  action: 'complete-forgot',
  phone,
  code,
  token: sent.token,
  password: newPassword,
})

await supabase.auth.setSession({
  access_token: result.access_token!,
  refresh_token: result.refresh_token!,
})
```

服务端只校验密码至少 8 位。食鉴前端额外要求：

- 8–20 位；
- 同时包含字母和数字；
- 不包含中文；
- 不使用常见弱密码；
- 不以手机号后 6 位结尾。

新项目应在服务端复刻完整密码策略，不能只依赖前端校验。

## 12. Edge Function action 契约

| action | 请求字段 | 说明 |
| --- | --- | --- |
| `send` | `phone`, `purpose` | 发送验证码；`purpose` 为 `login`、`signup` 或 `forgot` |
| `verify` | `phone`, `code`, `token` | 只验证验证码，不创建 Session |
| `complete-login` | `phone`, `code`, `token` | 验证码登录；手机号不存在时自动注册 |
| `password-login` | `phone`, `password` | 手机号密码登录 |
| `complete-signup` | `phone`, `code`, `token`, `password`, `nickname?` | 显式注册；后端已支持，但当前食鉴 UI 未使用 |
| `complete-forgot` | `phone`, `code`, `token`, `password` | 重置密码并登录 |

成功登录响应：

```json
{
  "ok": true,
  "access_token": "<jwt>",
  "refresh_token": "<refresh-token>",
  "expires_in": 3600,
  "token_type": "bearer"
}
```

错误响应：

```json
{
  "ok": false,
  "error": "可展示或转换的错误信息"
}
```

调用方必须以 `ok` 和必需字段为准，不能只根据 HTTP 200 判断业务成功。当前函数部分业务错误仍使用 HTTP 200。

## 13. 当前限流与超时

当前 Edge Function 使用进程内 `Map` 限流：

| 操作 | 维度 | 限制 |
| --- | --- | --- |
| 发送验证码 | 手机号 + purpose | 3 次/分钟 |
| 发送验证码 | 手机号 + purpose | 8 次/小时 |
| 发送验证码 | IP | 30 次/小时 |
| 密码登录 | 手机号 + IP | 8 次/15 分钟 |
| 密码登录 | IP | 60 次/15 分钟 |
| 校验 OTP | 手机号 + IP | 12 次/5 分钟 |
| 校验 OTP | IP | 80 次/5 分钟 |

其他时间参数：

- 阿里云请求超时：10 秒；
- 前端发送请求超时：15 秒；
- 前端重发倒计时：60 秒；
- OTP token 有效期：5 分钟。

进程内限流只适合基础防护。Edge Function 多实例、重启或冷启动后，各实例计数不共享且会重置。正式生产项目应将限流迁移到 Redis、Upstash、数据库原子计数器或网关限流。

## 14. Session 恢复

应用启动时至少执行：

```ts
const { data } = await supabase.auth.getSession()
setSession(data.session)

const { data: listener } = supabase.auth.onAuthStateChange((_event, session) => {
  setSession(session)
})

// 组件卸载时
listener.subscription.unsubscribe()
```

路由守卫应在 Session 初始化完成后再判断是否跳转登录页，避免刷新时短暂误判未登录。

食鉴开发环境还支持匿名/fixture 自动登录。它们不是短信登录必需配置，迁移生产功能时不要复制：

- `VITE_AUTH_LAX_DEV`
- `VITE_AUTH_STRICT_AUTH`
- `VITE_FIXTURE_AUTO_LOGIN`
- `VITE_FIXTURE_EMAIL`
- `VITE_FIXTURE_PASSWORD`
- `VITE_ENABLE_EMAIL_AUTH`

## 15. 上线前必须处理的安全项

当前方案可运行，但直接迁移到高流量或高风险生产环境前，应完成：

1. 将进程内限流替换为共享存储限流；
2. 接入图形验证码、滑块或设备风控，至少保护 `send`；
3. 将 `Math.random()` 生成 OTP 改为 Web Crypto 安全随机数；
4. 增加 OTP nonce 一次性消费记录，成功后立即失效，防止 5 分钟内重放；
5. 使用恒定时间比较 HMAC；
6. 收紧 CORS，避免无关站点直接调用；
7. 将前端密码规则同步到服务端；
8. 避免“忘记密码发送阶段”直接暴露手机号是否注册，降低账号枚举风险；
9. 将 `findUserByPhoneOrEmail()` 的分页扫描替换为索引化手机号身份映射；当前实现最多扫描 20 页、每页 1000 人；
10. 给 RAM 用户最小权限，不使用阿里云主账号 AccessKey；
11. 对发送量、失败率、429、阿里云错误码和函数 5xx 建立告警；
12. 定期轮换阿里云 AccessKey 和 `ALIYUN_OTP_HMAC_SECRET`；
13. 日志中不得记录验证码、AccessKey、Session token 或完整手机号；
14. 评估当前“找到手机号后覆盖用户 email 为合成邮箱”的行为是否符合新项目账号模型。

轮换 `ALIYUN_OTP_HMAC_SECRET` 会让尚未过期的旧 OTP token 立即失效，建议在低峰执行。

## 16. 验收步骤

### 16.1 部署检查

```bash
supabase functions list
supabase secrets list
```

确认：

- `sms-otp` 为 `ACTIVE`；
- 所有阿里云 Secret 名称存在；
- `verify_jwt = false`；
- 前端 URL、Anon Key 和函数属于同一个 Supabase 项目。

### 16.2 功能用例

至少逐项测试：

1. 新手机号发送验证码成功；
2. 错误验证码被拒绝；
3. 超过 5 分钟的验证码被拒绝；
4. 新手机号验证码登录后自动创建 Auth 用户；
5. `auth.users.phone` 为 `+86` E.164 格式；
6. 若启用 `profiles`，对应 profile 自动创建且手机号一致；
7. 退出后能再次用验证码登录；
8. 完成设置密码后可用手机号 + 密码登录；
9. 已注册手机号可重置密码并自动登录；
10. 未注册手机号不能走忘记密码；
11. 60 秒内前端不可重复发送；
12. 服务端超过频率后返回 429 和 `Retry-After`；
13. 刷新页面后 Session 仍能恢复；
14. Access token 到期后可自动刷新；
15. Web、iOS、Android 各自网络环境均能访问 Supabase Functions 和阿里云。

### 16.3 数据检查

在 Supabase Dashboard 中确认：

- Authentication → Users 中只有一个对应手机号用户；
- phone 已确认；
- 合成邮箱与手机号确定性映射一致；
- 没有因 `+86`、`86`、国内 11 位格式差异创建重复账号；
- `profiles.id = auth.users.id`；
- RLS 策略不会允许普通用户读取其他用户手机号。

## 17. 常见故障

### `未配置环境变量：ALIYUN_...`

对应 Edge Function Secret 缺失。执行 `supabase secrets list` 检查名称，补齐后重新部署函数。

### `FunctionsFetchError`

依次检查：

- `sms-otp` 是否已部署；
- 前端 URL 和 Anon Key 是否属于同一项目；
- 函数是否允许未登录请求；
- 浏览器、App 或所在网络能否访问 Supabase Functions；
- CORS 是否包含当前 origin。

### `SignatureDoesNotMatch`

检查：

- AccessKey ID/Secret 是否配对；
- 阿里云 Secret 是否带多余空格或换行；
- 请求时间是否准确；
- 签名名 Base64 解码后是否正确；
- RAM 用户是否有对应 API 权限。

### `短信模板不存在或未审核`

检查 `ALIYUN_SMS_TEMPLATE_CODE`，并确认它属于当前 AccessKey 所在账号及正确的阿里云产品。

### `短信签名不存在或未审核`

重新生成 `ALIYUN_SMS_SIGN_NAME_B64`，确认签名文字与控制台完全一致。

### 验证码正确但登录失败

检查：

- `SUPABASE_SERVICE_ROLE_KEY` 是否正确；
- Auth 用户是否已有冲突手机号或邮箱；
- synthetic email 域名是否在迁移中被改过；
- Edge Function 日志中的 Auth 管理接口错误；
- Supabase Auth 邮箱身份功能是否可用。

### 用户量增大后登录变慢

当前按页调用 `admin.auth.admin.listUsers()` 查找手机号，复杂度随用户量增长。应建立 `phone_identities(phone_e164 primary key, user_id uuid unique)` 映射表，通过索引直接查询。
